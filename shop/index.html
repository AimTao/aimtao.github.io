md5 盐值加密用户密码

对密码储存的要求：不储存密码明文，对密码加密后，对密文使用 md5 摘要算法，使密文不可反解。

加盐

  • 生成随机数作为盐值 salt，将 salt 和密码组合成字符串。对字符串进行加密后，对密文使用 md5 摘要算法。
  • 数据库同时储存 md5 和 salt，验证密码时，将 salt 和密码拼接为字符串，使用同样的加密算法加密后，生成 md5 比对即可。

为什么要加盐？

为了增加 md5 值被反解的难度。即使数据库泄漏、密钥泄漏、加密代码被逆向，攻击者也无法知道用户的原始密码。

  • 字符串（通常是加密后的密文）太短/太简单，生成的 md5 容易被彩虹表进行反解出字符串。
  • 在字符串中加入 salt，可以保证新的字符串足够长、随机，这样生成的 md5 值不容易被彩虹表反解。

加盐就一定安全吗？

不一定，为了保证安全，（也就是增加攻击者的反解成本），起码需要做到两点：

  • salt 不能太短：salt + 字符串生成的新字符串不够长不够随机，就可能直接被反解出来。
  • salt 不使用固定值，要保证随机生成：如果 salt 泄漏，攻击者可根据该 salt，专门跑一遍程序，生成针对该 salt 的彩虹表，依然可以反解。但当每个用户 salt 值不同，均需要生成对应的彩虹表，反解成本高。

负载均衡

必须导入 grpc-consul-resolver，这里面实现了一个 consul 的解析器，用来进行 grpc 的服务发现。
在哪里使用了？导入时，会自动执行 grpc-consul-resolver 中的 consul.init 方法，init 方法注册了 consul 解析器，解析器会解析 grpc.Dail 的 target，并请求 consul 服务器，获取指定的服务。
稍微展开一下，consul.init 在调用 resolver.Register(&builder{}) 时，先实例化了一个自定义的 builder，并注册到 rpc 的 resolve 中。当 grpc 执行 build 的时，发现 target 前面是字符串 “consul”，就会使用这个自定义的 builder 实例的 Build 方法。在这个 Build 方法里，请求 consul 服务器，获取指定的服务 list。

负载均衡拨号时，target 是 consul 的地址，并使用 grpc 的负载均衡器。
